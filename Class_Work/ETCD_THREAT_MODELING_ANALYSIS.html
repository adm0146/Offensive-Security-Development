ü§ñ Claude's Threat Analysis

CRITICAL

Threat 1: Unauthenticated etcd API Access

Threat Description

An attacker exploits an etcd cluster configured without authentication, gaining direct read/write access to the key-value store via gRPC or HTTP API.

Threat Actor

  • External attacker with network access
  • Malicious insider on same network segment
  • Compromised adjacent workload in cluster

Preconditions

  • etcd client port (2379) reachable from attacker's network
  • Authentication disabled (--client-cert-auth=false)
  • No network segmentation isolating etcd

Impact

  • Confidentiality: Complete exposure of all Kubernetes secrets
  • Integrity: Ability to modify cluster state
  • Availability: Cluster can be rendered inoperable

Post-Conditions

  • Attacker possesses all cluster credentials
  • Persistent backdoor access possible
  • Full cluster compromise achievable
HIGH

Threat 2: etcd Peer Communication MITM Attack

Threat Description

An attacker with network positioning between etcd members performs MITM on peer communication (port 2380), intercepting Raft protocol messages.

Threat Actor

  • Compromised network device (switch/router)
  • Cloud provider insider
  • ARP spoofing attacker on shared network

Preconditions

  • Peer TLS disabled (--peer-client-cert-auth=false)
  • Network positioning between etcd nodes
  • Certificate validation disabled

Impact

  • All replicated data visible in transit
  • Potential consensus manipulation
  • Cluster stability compromised
HIGH

Threat 3: Snapshot/Backup Data Exfiltration

Threat Description

An attacker gains access to etcd snapshot files containing complete database state with all secrets in portable format.

Key Points

  • Snapshots stored without encryption at rest
  • Backup storage permissions overly permissive
  • Complete extraction of all historical secrets
HIGH

Threat 4: Resource Exhaustion DoS

Threat Description

Attacker launches DoS via thousands of concurrent watch connections or expensive range queries, exhausting memory/CPU/file descriptors.

Impact

  • Complete Kubernetes control plane outage
  • Scheduler, controller-manager unable to function
  • Manual intervention required to restore
CRITICAL

Threat 5: Client Certificate Private Key Compromise

Threat Description

Attacker obtains etcd client certificate private keys through filesystem access, compromise of cert systems, or IaC repositories.

Key Points

  • Provides authenticated access indistinguishable from legitimate operations
  • Persistent access until certificates rotated/revoked
  • Difficult to distinguish malicious from legitimate in audit logs

ü§ñ Gemini's Threat Analysis (STRIDE-Based)

Gemini provides a STRIDE-methodology approach to threat modeling for etcd:

CRITICAL

Threat 1: Rogue Peer Join

Category: Spoofing / Information Disclosure

An attacker "joins" a rogue node to the cluster by exploiting unauthenticated peer communication, synchronizing the entire keyspace.

  • Maintains persistent, live-updating mirror of cluster state
  • Full copy of all cluster data including secrets and tokens
HIGH

Threat 2: Plaintext Client Traffic

Category: Information Disclosure

Unencrypted client-to-server communication exposes sensitive data sent by API server (Secret objects) in plaintext.

HIGH

Threat 3: Cold Boot/Disk Access

Category: Information Disclosure / Tampering

Attacker with disk access can use etcdutl to dump entire database offline, bypassing all etcd-level RBAC and authentication.

MEDIUM

Threat 4: Storage DoS

Category: Denial of Service

Resource exhaustion via large volume of expensive range requests or high-frequency writes.

  • Puts cluster into read-only state
  • Requires defragmentation and manual intervention
HIGH

Threat 5: RBAC Misconfiguration

Category: Elevation of Privilege

Insider with overly broad etcd RBAC permissions can modify ClusterRoleBinding to grant cluster-admin rights.

üõ°Ô∏è Recommended Mitigations

üîê Authentication & Authorization

  • Enable --client-cert-auth=true
  • Enable --peer-client-cert-auth=true
  • Implement etcd RBAC
  • Use strong, rotated credentials

üîí Encryption

  • Enforce TLS for all client connections (2379)
  • Enforce mTLS for peer communication (2380)
  • Enable encryption at rest
  • Encrypt snapshots and backups

üîë Credential Management

  • Store keys in HSM or secrets manager
  • Implement certificate rotation
  • Audit all certificate access
  • Use automated cert management tools

üìä Resource Management

  • Implement rate limiting
  • Set appropriate storage quotas
  • Monitor resource consumption
  • Set connection limits

üíæ Physical/Storage Security

  • Encrypt etcd volumes at rest
  • Restrict physical access to etcd nodes
  • Use hardware security modules
  • Implement secure boot and attestation

üåê Network Security

  • Isolate etcd from untrusted networks
  • Use network policies and firewalls
  • Disable unnecessary exposure
  • Monitor network access
